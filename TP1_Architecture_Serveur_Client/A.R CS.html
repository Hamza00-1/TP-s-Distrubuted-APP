A — Communication distribuée : panorama API Fiabilité

A.1 — Appels synchrones vs asynchrones

Dans un système distribué, deux grands paradigmes de communication coexistent :

Critère Synchrone (Request / Response) Asynchrone (Messaging)
Principe Le client envoie une requête et attend la réponse du serveur. L'émetteur envoie un message et continue sans attendre de réponse immédiate.
Protocole type HTTP/HTTPS, gRPC (unary) AMQP, MQTT, Redis Pub/Sub, files de messages
Couplage temporel Fort — les deux parties doivent être disponibles en même temps Faible — le producteur et le consommateur peuvent fonctionner indépendamment
Couplage spatial Fort — le client doit connaître l'adresse du serveur Faible — communication via un broker intermédiaire
Latence perçue Directe (bloquante) Indirecte (non bloquante)
Complexité Simple à comprendre, simple à debugger Plus complexe : ordering, delivery guarantees, dead letters
Cas d'usage Lecture de données, opérations CRUD, APIs publiques Événements, notifications, traitements longs, découplage fort
📌 À retenir
Il n'y a pas de "meilleur" modèle. Le choix dépend du couplage acceptable, de la tolérance à la latence et de la criticité de la réponse immédiate. De nombreux systèmes réels combinent les deux.
Figure 1 — Comparaison Request/Response synchrone vs Messaging asynchrone (avec file de messages).

A.2 — Les trois types de couplage

Comprendre le couplage est essentiel pour arbitrer les choix d'architecture :

Type de couplage Définition Exemple fort Exemple faible
Temporel Les deux parties doivent être actives au même instant Appel HTTP direct Message dans une queue
Spatial L'émetteur doit connaître l'adresse exacte du récepteur URL en dur dans le code Service discovery, broker
Sémantique Les deux parties partagent un contrat de données (schéma) Structures internes partagées Contrat d'API versionné
⚠️ Piège classique
« On va tout passer en asynchrone pour découpler ! » — Attention : le messaging ajoute de la complexité (ordering, idempotence, dead-letter queues, monitoring). Chaque pattern résout un problème et en crée d'autres. Choisissez en conscience.

A.3 — Problèmes fondamentaux du réseau

Le réseau est intrinsèquement non fiable. Les « 8 erreurs de l'informatique distribuée » (Peter Deutsch) rappellent qu'on ne peut jamais supposer que le réseau est fiable, sans latence, avec une bande passante infinie, etc. Voici les problèmes concrets :

Problème Description Conséquence Analogie
Latence Délai entre l'envoi et la réception Ralentissement, timeouts, mauvaise UX Un courrier qui met 3 jours au lieu de 1
Gigue (jitter) Variation de la latence d'un paquet à l'autre Difficile de fixer un timeout optimal Certains courriers arrivent en 1 jour, d'autres en 5
Perte de paquets Des messages n'arrivent jamais Données manquantes, incohérences Courrier perdu par la poste
Duplication Un message arrive deux fois Double traitement (double débit, double inscription…) Deux exemplaires du même courrier livrés
Désordre Les messages arrivent dans un ordre différent de l'envoi Incohérence d'état, conflits de données Le chapitre 3 du livre arrive avant le chapitre 1
Partition réseau Deux sous-groupes ne peuvent plus communiquer Split-brain, décisions contradictoires Deux bureaux de poste isolés par une tempête
📌 À retenir
« Le réseau est fiable » est la première fausse hypothèse de l'informatique distribuée. Toute conception d'API et de client doit intégrer la possibilité de défaillance réseau.

B — Conception d'API (niveau professionnel) API

B.1 — Le contrat d'API

Une API (Application Programming Interface) est un contrat entre un fournisseur de service et ses consommateurs. Ce contrat définit précisément :

  • Endpoints : les URL accessibles (ex : /api/v1/documents)
  • Méthodes HTTP : GET, POST, PUT, DELETE, PATCH — chacune avec une sémantique précise
  • Payloads : structure JSON des requêtes et réponses
  • Codes de retour : 200, 201, 400, 401, 403, 404, 409, 429, 500, 503…
  • Headers : Content-Type, Authorization, X-Request-Id…
  • Comportements spéciaux : pagination, tri, filtrage, rate limiting
Méthode HTTP Sémantique Idempotente ? Corps de requête ? Utilisation typique
GET Lire une ressource ✅ Oui Non (paramètres dans l'URL) Récupérer un document, une liste
POST Créer une ressource ❌ Non (par défaut) Oui Créer un document, soumettre un formulaire
PUT Remplacer une ressource entière ✅ Oui Oui Mettre à jour un document (remplacement total)
PATCH Modifier partiellement une ressource ⚠️ Dépend Oui Mettre à jour un champ spécifique
DELETE Supprimer une ressource ✅ Oui Rarement Supprimer un document

B.2 — Codes d'erreur HTTP : les connaître par cœur

Code Signification Quand l'utiliser
200OKRequête réussie (GET, PUT, PATCH)
201CreatedRessource créée (POST réussi)
204No ContentSuccès sans corps de réponse (DELETE)
400Bad RequestPayload invalide, champ manquant, format incorrect
401UnauthorizedAuthentification manquante ou invalide
403ForbiddenAuthentifié mais pas autorisé
404Not FoundRessource inexistante
409ConflictConflit d'état (ressource déjà existante, version conflictuelle)
429Too Many RequestsRate limiting déclenché
500Internal Server ErrorErreur côté serveur (bug, crash)
502Bad GatewayLe serveur intermédiaire a reçu une réponse invalide
503Service UnavailableService temporairement indisponible (maintenance, surcharge)
504Gateway TimeoutLe service en amont n'a pas répondu à temps
⚠️ Piège classique
Ne renvoyez jamais 200 OK avec un corps contenant {"error": "..."}. Utilisez les codes HTTP pour ce qu'ils sont. Un client automatisé se fie au code de statut, pas au contenu du body.

B.3 — Validation d'entrée (JSON)

Toute donnée reçue de l'extérieur est potentiellement malveillante ou malformée. La validation d'entrée est une obligation, pas une option :

  • Vérifier la présence des champs obligatoires
  • Vérifier les types (string, int, bool, array)
  • Vérifier les contraintes (longueur max, format email, plage numérique)
  • Rejeter l'inconnu : des champs inattendus peuvent être une tentative d'injection
💡 Exemple concret
Un endpoint POST /documents attend : 
{
  "title": "string, requis, 1-200 caractères",
  "content": "string, requis, max 50000 caractères",
  "tags": "array de strings, optionnel, max 10 éléments",
  "visibility": "string, requis, valeurs: 'public' | 'private' | 'team'"
}
Si title est absent → 400. Si visibility vaut "admin_override" → 400. Aucun traitement ne doit commencer avant validation complète.

B.4 — Versioning d'API

Les APIs évoluent. Comment gérer les changements sans casser les clients existants ?

Stratégie Exemple Avantage Inconvénient
URI path /api/v1/documents → /api/v2/documents Explicite, facile à comprendre et router Duplication de code, URLs longues
Header Accept: application/vnd.myapp.v2+json URL propre, séparation contenu/version Moins visible, plus complexe à tester
Query param /api/documents?version=2 Simple à ajouter Pollue l'URL, mélange version/données
📌 À retenir
La stratégie URI path (/api/v1/) est la plus courante et la plus recommandée pour les APIs publiques. Quelle que soit la stratégie, documentez-la clairement et maintenez la rétrocompatibilité pendant une durée définie.

B.5 — Idempotence

Définition : Une opération est idempotente si l'appeler N fois produit le même résultat que l'appeler 1 fois. Le système converge vers le même état.

Opération Idempotente ? Explication
GET /documents/42 ✅ Oui Lire 10 fois ne change rien à l'état
PUT /documents/42 (remplacement) ✅ Oui Remplacer 10 fois par les mêmes données → même résultat
DELETE /documents/42 ✅ Oui Supprimer un document déjà supprimé → pas de changement
POST /documents ❌ Non Chaque appel crée un nouveau document → duplication !
POST /documents avec Idempotency-Key ✅ Oui (si implémenté) Le serveur détecte la duplication grâce à la clé unique
⚠️ Piège classique
POST /account/debit {"amount": 100} — Si le réseau est lent et que le client retente, le compte est débité deux fois. Solution : envoyer un Idempotency-Key: abc-123 que le serveur stocke. Si la même clé revient, il retourne la réponse précédente sans exécuter l'opération à nouveau.

B.6 — Pagination, filtrage, tri

Quand une collection contient des milliers d'éléments, il est impensable de tout retourner d'un coup. Trois mécanismes complémentaires :

  • Pagination : GET /documents?page=2&per_page=20 — Retourner les résultats par lots. Inclure dans la réponse : total, page, per_page, total_pages.
  • Filtrage : GET /documents?status=published&author=alice — Ne retourner que les éléments correspondants.
  • Tri : GET /documents?sort=created_at&order=desc — Contrôler l'ordre des résultats.
🔒 Point cybersécurité
Pagination obligatoire : sans elle, un attaquant peut faire GET /documents?per_page=999999 et vider la mémoire du serveur (DoS). Imposez un per_page maximum (ex : 100) côté serveur, quelle que soit la valeur demandée par le client.

C — Fiabilité et résilience côté client Fiabilité

C.1 — Timeouts : pourquoi ils sont indispensables

Un timeout est une durée maximale au-delà de laquelle le client considère que la requête a échoué. Sans timeout :

  • Le thread/processus client reste bloqué indéfiniment
  • Les ressources (sockets, mémoire, connexions) s'accumulent → famine de ressources
  • Un service lent contamine les services appelants → effet cascade

Types de timeouts :

Type Description Valeur typique
Connection timeout Temps max pour établir la connexion TCP 3-5 secondes
Read timeout Temps max pour recevoir la réponse après connexion 5-30 secondes (selon l'opération)
Total timeout Durée totale de l'opération incluant retries 30-120 secondes
📌 À retenir
Toute requête réseau doit avoir un timeout. C'est une règle absolue en systèmes distribués. Un timeout trop court cause des faux échecs ; un timeout trop long cause des blocages. La juste valeur dépend du SLA du service appelé.

C.2 — Retries : quand c'est utile vs dangereux

Retenter une requête échouée peut résoudre des problèmes transitoires (glitch réseau, redémarrage de service). Mais des retries mal gérés sont dangereux :

Situation Retry utile ? Explication
Erreur 503 (service indisponible) ✅ Oui Problème transitoire, retry après attente
Erreur 500 (erreur serveur) ⚠️ Dépend Si le serveur a un bug, retenter n'aide pas
Erreur 429 (rate limited) ✅ Oui, après attente Respecter le Retry-After header
Erreur 400 (bad request) ❌ Non La requête est invalide, retenter est inutile
Erreur 401 (non authentifié) ❌ Non (sauf refresh token) Credentials invalides, retenter identiquement est inutile
Timeout sur POST /payment ⚠️ Dangereux ! Le serveur a peut-être traité la requête → double paiement !
⚠️ Piège classique — Retry Storm
Si un service tombe et que 1000 clients retentent tous en même temps, le service reçoit un pic de charge dès qu'il redémarre et retombe immédiatement. C'est l'amplification par retry. Solutions : backoff + jitter + circuit breaker.

C.3 — Backoff exponentiel + jitter

Le backoff exponentiel consiste à augmenter le délai entre chaque retry :

délai = min(base_delay × 2^tentative, max_delay)

Exemple avec base_delay = 1s, max_delay = 30s :
  Tentative 0 → 1s
  Tentative 1 → 2s
  Tentative 2 → 4s
  Tentative 3 → 8s
  Tentative 4 → 16s
  Tentative 5 → 30s (plafonné)

Le jitter ajoute un composant aléatoire pour éviter que tous les clients retentent au même instant :

délai_final = délai × random(0.5, 1.5)

Ou "full jitter" :
délai_final = random(0, délai)
Figure 2 — Timeline d'un backoff exponentiel avec jitter : chaque retry attend de plus en plus longtemps, avec une variation aléatoire.

C.4 — Circuit Breaker (concept)

Le circuit breaker est un pattern inspiré de l'électricité : quand un service échoue trop souvent, le « disjoncteur » s'ouvre et les requêtes sont immédiatement rejetées sans même être envoyées. Cela protège le service malade et évite l'effet cascade.

Figure 3 — Circuit Breaker : trois états (Fermé → Ouvert → Semi-ouvert) et leurs transitions.
📌 À retenir
Le circuit breaker protège le système globalement : il évite de surcharger un service déjà en difficulté, libère les ressources du client et permet une reprise progressive via l'état semi-ouvert.

C.5 — Bulkhead et Fallback (concepts)

Bulkhead (cloison étanche) : Inspiré des cloisons de navires, ce pattern isole les différents appels de service dans des « compartiments » séparés (pools de threads/connexions distincts). Si le service A est lent, il ne consomme pas les ressources allouées au service B.

Fallback (repli) : Quand un appel échoue (même après retries), le système fournit une réponse dégradée plutôt qu'une erreur brute. Exemples :

  • Retourner des données en cache (même périmées)
  • Retourner une valeur par défaut
  • Rediriger vers un service de secours
  • Informer l'utilisateur clairement (« Service temporairement indisponible, réessayez dans 5 minutes »)

C.6 — Tableau récapitulatif : problèmes réseau et réponses

Problème réseau Symptôme Réponse recommandée Risque si mal géré
Service lent Latence élevée, timeouts fréquents Timeout + backoff + circuit breaker Effet cascade, blocage de threads
Service intermittent Erreurs 503 aléatoires Retry avec backoff + jitter Retry storm si pas de jitter
Service en panne totale Connexion refusée / timeout immédiat Circuit breaker + fallback Cascade de pannes sur tous les appelants
Réseau partitionné Certains nœuds joignables, d'autres non Détection + basculement + logs Split-brain, incohérences de données
Duplication de requêtes Actions exécutées deux fois Idempotency key + déduplication serveur Double paiement, données corrompues
Surcharge (rate limit) Erreurs 429 Respecter Retry-After, backoff Bannissement temporaire / permanent
Désordre des messages Événements traités dans le mauvais ordre Numéros de séquence, timestamps, buffers État incohérent

D — Observabilité minimale Concept

D.1 — Le problème du debug distribué

En local, on peut poser un breakpoint, lire un fichier de log, reproduire le bug. En distribué, tout change :

Aspect Debug local Debug distribué
Reproduction Facile (même machine, même état) Très difficile (état réparti, timing, charge)
Logs Un seul fichier, chronologique N fichiers sur N machines, horloges désynchronisées
Causalité Ordre d'exécution clair Appels entre services, concurrence, race conditions
Outils Debugger, print, log local Tracing distribué, métriques, alertes, correlation IDs

D.2 — Les trois piliers de l'observabilité

Logs structurés

Plutôt que print("Erreur !"), utilisez des logs structurés en JSON qui incluent un contexte riche :

{
"timestamp": "2026-03-09T14:32:01.234Z",
"level": "ERROR",
"service": "document-service",
"request_id": "req-a1b2c3d4",
"user_id": "user-42",
"method": "POST",
"path": "/api/v1/documents",
"status": 500,
"duration_ms": 342,
"error": "DatabaseConnectionError: timeout after 5000ms",
"trace_id": "trace-xyz-789"
}

Le champ request_id (ou correlation_id) est crucial : il permet de suivre une requête utilisateur à travers tous les services qu'elle traverse.

Traces distribuées

Une trace représente le parcours complet d'une requête à travers le système. Chaque service ajoute un span (segment) à la trace. Cela permet de visualiser :

  • Quel service a été appelé, dans quel ordre
  • Combien de temps chaque service a pris
  • Où se situe le goulot d'étranglement
┌─ Trace: req-a1b2c3d4 ───────────────────────────────────────────┐
│                                                                  │
│  [API Gateway]  ──── 12ms ────┐                                  │
│                                │                                  │
│       [Auth Service]  ── 3ms ──┘── 45ms ────┐                    │
│                                              │                    │
│            [Document Service]  ── 280ms ─────┘                    │
│                   │                                               │
│                   └── [Database Query] ── 250ms (⚠️ lent !)      │
│                                                                  │
└──────────────── Durée totale : 342ms ────────────────────────────┘

Métriques

Les métriques sont des mesures numériques agrégées dans le temps. Les plus importantes pour les APIs :

  • Latence : temps de réponse — mesurée en percentiles (p50, p95, p99)
    • p50 = 50% des requêtes sont plus rapides que cette valeur (médiane)
    • p95 = 95% des requêtes sont plus rapides
    • p99 = 99% des requêtes sont plus rapides (les « queues » de distribution)
  • Taux d'erreur : pourcentage de réponses 5xx
  • Débit : requêtes par seconde (RPS)
  • Saturation : utilisation CPU, mémoire, connexions DB
💡 Exemple concret
« Notre API a un p50 de 45ms mais un p99 de 2.3s. » Cela signifie que la plupart des requêtes sont rapides, mais 1% met plus de 2 secondes. Ce 1% peut représenter des milliers d'utilisateurs mécontents par jour. On ne peut pas se contenter de la moyenne !
📌 À retenir
Bonnes pratiques d'observabilité :
  • Générer un X-Request-Id dès l'entrée et le propager à chaque service
  • Logger en JSON structuré (jamais en texte libre non parsable)
  • Mesurer la latence en percentiles, pas en moyenne
  • Alerter sur les anomalies (taux d'erreur > 1%, p99 > seuil)
  • Ne jamais logger de données sensibles (mots de passe, tokens, numéros de carte)

E — Implications cybersécurité des APIs Sécurité

E.1 — AuthN vs AuthZ : clarification fondamentale

Concept Authentification (AuthN) Autorisation (AuthZ)
Question « Qui es-tu ? » « As-tu le droit de faire ça ? »
Mécanisme Login/mot de passe, token, certificat Rôles (RBAC), attributs (ABAC), politiques
Échec Code 401 Unauthorized Code 403 Forbidden
Analogie Montrer son badge à l'entrée Le badge donne accès au 2e étage, pas au 5e
Ordre Toujours en premier Après l'authentification

E.2 — Gestion des secrets

Un secret est toute donnée qui, si elle est divulguée, compromet la sécurité du système : clés API, mots de passe de base de données, clés de chiffrement, tokens de service.

Pratique ❌ Mauvais ✅ Bon
Stockage En dur dans le code source Variables d'environnement ou vault sécurisé
Versioning Commité dans Git Exclu via .gitignore, injecté à l'exécution
Rotation Jamais changé depuis 3 ans Rotation régulière (90 jours max), automatisée
Accès Même token pour tous les services Un token par service, principe du moindre privilège
Logging Token dans les logs applicatifs Masqué / jamais loggé

E.3 — Panorama des attaques sur les APIs

Menace Impact Indices de détection Contre-mesure
MITM (Man-in-the-Middle) Interception et modification des données en transit Certificats invalides, alertes TLS, données altérées TLS/HTTPS obligatoire, certificate pinning
Replay attack Rejouer une requête légitime pour duplicer une action Actions dupliquées, timestamps incohérents Nonces, timestamps + fenêtre de validité, idempotency keys
Injection JSON Injecter des données malveillantes dans les payloads Champs inattendus, caractères spéciaux, erreurs de parsing Validation stricte d'entrée, whitelisting des champs
Brute force Deviner credentials par essais massifs Volume anormal de 401, mêmes endpoints auth Rate limiting, verrouillage progressif, CAPTCHA
DDoS applicatif Saturer le service avec des requêtes légitimes coûteuses Pic de charge, latence explosive, CPU 100% Rate limiting, pagination forcée, circuit breaker
SSRF (Server-Side Request Forgery) Forcer le serveur à faire des requêtes vers des services internes Requêtes vers localhost, IPs privées, métadonnées cloud Validation d'URL, blocage des IPs internes, DNS rebinding protection
Enumeration Découvrir des ressources existantes (users, documents) Parcours séquentiel d'IDs, timing differences UUIDs au lieu d'IDs séquentiels, réponses uniformes (timing constant)
Token theft Vol de token d'authentification Utilisation depuis des IPs/geos inhabituelles Expiration courte, refresh tokens, binding IP/device

E.4 — Modèles d'autorisation : RBAC vs ABAC

Modèle Principe Exemple Usage idéal
RBAC (Role-Based Access Control) Les permissions sont attribuées à des rôles, les utilisateurs reçoivent des rôles Rôle « éditeur » → peut créer/modifier des documents Systèmes avec rôles bien définis et stables
ABAC (Attribute-Based Access Control) Les permissions dépendent d'attributs (user, ressource, contexte) « Un utilisateur du département RH peut accéder aux fiches du département RH pendant les heures de bureau » Systèmes avec règles complexes et contextuelles
🔒 Point cybersécurité : pourquoi « internal network » ≠ confiance

Une erreur très courante est de considérer que les communications entre services internes n'ont pas besoin de sécurité : « c'est sur notre réseau privé, donc c'est sûr ».

C'est faux pour de multiples raisons :

  • Mouvement latéral : un attaquant qui compromet un service peut pivoter vers tous les autres services « de confiance »
  • Insider threat : un employé malveillant ou négligent a accès au réseau interne
  • Supply chain : une dépendance compromise peut envoyer des requêtes internes malveillantes
  • Erreur de configuration : un service interne exposé par erreur sur Internet

Principe Zero Trust : vérifier chaque requête, même interne. Authentifier les services entre eux (mTLS, tokens de service), valider les entrées, appliquer le moindre privilège.

E.5 — Rate Limiting

Le rate limiting limite le nombre de requêtes qu'un client peut effectuer dans une fenêtre de temps donnée. C'est à la fois un mécanisme de fiabilité (protéger le serveur) et de sécurité (empêcher le brute force et le DDoS).

Algorithme Principe Avantage
Fixed window Compteur remis à zéro toutes les N secondes Simple à implémenter
Sliding window Fenêtre glissante (pas de reset brutal) Plus lisse, moins d'effets de bord
Token bucket Un seau se remplit à débit constant ; chaque requête consomme un token Permet des bursts contrôlés

Headers standard à retourner quand le rate limit est atteint :

HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1741531800

Live coding Python (45 min) API Fiabilité

Trois mini-exemples commentés ligne par ligne. On utilise uniquement la bibliothèque standard Python (http.server, urllib, json, time, random).

Exemple 1 — Mini API HTTP avec http.server

Ce serveur expose deux endpoints : GET /health et POST /documents. Il gère les codes d'erreur 400, 401, 404 et 500.

import json
import uuid
from http.server import HTTPServer, BaseHTTPRequestHandler
from datetime import datetime

# ── Stockage en mémoire (simulation d'une base de données) ──
documents_db = {}

# ── Token attendu (en production : JAMAIS en dur !) ──
VALID_TOKEN = "secret-token-abc123"


class APIHandler(BaseHTTPRequestHandler):
  """Gestionnaire de requêtes HTTP pour notre mini API."""

  def _send_json(self, status_code, data):
      """Envoie une réponse JSON avec le code de statut donné."""
      self.send_response(status_code)                # Code HTTP (200, 400…)
      self.send_header("Content-Type",
                       "application/json")        # Type MIME
      self.end_headers()                             # Fin des headers
      self.wfile.write(                              # Écriture du body
          json.dumps(data).encode("utf-8")
      )

  def _check_auth(self):
      """Vérifie le header Authorization. Retourne True si valide."""
      auth = self.headers.get("Authorization", "")
      # Format attendu : "Bearer secret-token-abc123"
      if auth != f"Bearer {VALID_TOKEN}":
          self._send_json(401, {
              "error": "unauthorized",
              "message": "Token manquant ou invalide"
          })
          return False
      return True

  # ── GET ────────────────────────────────────────────────────
  def do_GET(self):
      # Endpoint de santé : pas d'auth nécessaire
      if self.path == "/health":
          self._send_json(200, {
              "status": "ok",
              "timestamp": datetime.utcnow().isoformat(),
              "documents_count": len(documents_db)
          })
          return

      # Tout autre GET → 404
      self._send_json(404, {
          "error": "not_found",
          "message": f"Endpoint {self.path} inconnu"
      })

  # ── POST ───────────────────────────────────────────────────
  def do_POST(self):
      if self.path != "/documents":
          self._send_json(404, {
              "error": "not_found",
              "message": f"Endpoint {self.path} inconnu"
          })
          return

      # ① Vérifier l'authentification
      if not self._check_auth():
          return

      # ② Lire et parser le body JSON
      content_length = int(
          self.headers.get("Content-Length", 0)
      )
      if content_length == 0:
          self._send_json(400, {
              "error": "bad_request",
              "message": "Corps de requête vide"
          })
          return

      try:
          raw_body = self.rfile.read(content_length)
          body = json.loads(raw_body)
      except json.JSONDecodeError:
          self._send_json(400, {
              "error": "bad_request",
              "message": "JSON invalide"
          })
          return

      # ③ Validation des champs
      title = body.get("title", "").strip()
      content = body.get("content", "").strip()

      if not title or not content:
          self._send_json(400, {
              "error": "validation_error",
              "message": "'title' et 'content' sont requis"
          })
          return

      if len(title) > 200:
          self._send_json(400, {
              "error": "validation_error",
              "message": "'title' ne doit pas dépasser 200 caractères"
          })
          return

      # ④ Créer le document
      doc_id = str(uuid.uuid4())
      documents_db[doc_id] = {
          "id": doc_id,
          "title": title,
          "content": content,
          "created_at": datetime.utcnow().isoformat()
      }

      # ⑤ Répondre 201 Created
      self._send_json(201, documents_db[doc_id])

  # Supprimer les logs de requête par défaut (trop verbeux)
  def log_message(self, format, *args):
      pass


if __name__ == "__main__":
  server = HTTPServer(("127.0.0.1", 8080), APIHandler)
  print("Serveur démarré sur http://127.0.0.1:8080")
  server.serve_forever()

Exemple 2 — Client Python (urllib) avec timeout et X-Request-Id

import json
import uuid
from urllib.request import Request, urlopen
from urllib.error import HTTPError, URLError


def api_request(method, url, data=None, token=None,
              timeout=10):
  """
  Effectue une requête HTTP vers l'API.
  - method  : "GET" ou "POST"
  - url     : URL complète de l'endpoint
  - data    : dict Python (sera converti en JSON)
  - token   : token d'authentification
  - timeout : durée max en secondes (défaut : 10s)
  Retourne : (status_code, response_dict)
  """

  # ── Préparer le body JSON si nécessaire ──
  body_bytes = None
  if data is not None:
      body_bytes = json.dumps(data).encode("utf-8")

  # ── Construire la requête ──
  req = Request(url, data=body_bytes, method=method)
  req.add_header("Content-Type",
                 "application/json")

  # ── Ajouter un identifiant unique de requête ──
  # Permet la traçabilité dans les logs du serveur
  request_id = str(uuid.uuid4())
  req.add_header("X-Request-Id", request_id)
  print(f"[{request_id[:8]}] {method} {url}")

  # ── Ajouter le token si fourni ──
  if token:
      req.add_header("Authorization",
                     f"Bearer {token}")

  # ── Envoyer avec timeout ──
  try:
      with urlopen(req, timeout=timeout) as resp:
          body = json.loads(resp.read().decode("utf-8"))
          print(f"  ✅ {resp.status}")
          return resp.status, body

  except HTTPError as e:
      # Erreur HTTP (4xx, 5xx) : lire le body d'erreur
      error_body = {}
      try:
          error_body = json.loads(
              e.read().decode("utf-8")
          )
      except Exception:
          pass
      print(f"  ❌ HTTP {e.code}: {error_body}")
      return e.code, error_body

  except URLError as e:
      # Erreur réseau (connexion refusée, DNS, timeout…)
      print(f"  🔌 Erreur réseau: {e.reason}")
      return None, {"error": str(e.reason)}

💡 Exemple concret — Le X-Request-Id
Le client génère un UUID unique pour chaque requête et l'envoie via le header X-Request-Id. Le serveur le loggue. Si un problème survient, on peut retrouver la requête précise dans les logs de tous les services traversés. C'est le principe du correlation ID.

Exemple 3 — Retry avec backoff exponentiel

import time
import random


def request_with_retry(
  func,                       # fonction appelant l'API
  max_retries=4,              # nombre max de tentatives
  base_delay=1.0,             # délai initial (secondes)
  max_delay=30.0,             # délai maximum (plafond)
  retryable_statuses=(500, 502, 503, 504)
):
  """
  Appelle func() avec retry + backoff exponentiel + jitter.

  func() doit retourner (status_code, response_body).
  On ne retente QUE sur les codes de statut « retryables ».
  Les erreurs 4xx ne sont JAMAIS retentées (sauf 429).
  """

  for attempt in range(max_retries + 1):

      status, body = func()

      # ── Succès → retourner immédiatement ──
      if status is not None and status < 500 \
              and status != 429:
          return status, body

      # ── Rate limited (429) → respecter Retry-After ──
      if status == 429:
          # En production, lire le header Retry-After
          wait = 30.0
          print(f"  ⏳ Rate limited. Attente {wait}s…")
          time.sleep(wait)
          continue

      # ── Dernière tentative atteinte → abandonner ──
      if attempt == max_retries:
          print(f"  💀 Abandon après {max_retries+1} tentatives")
          return status, body

      # ── Calculer le délai de backoff exponentiel ──
      delay = min(base_delay * (2 ** attempt), max_delay)

      # ── Ajouter du jitter (variation aléatoire) ──
      # "Full jitter" : valeur entre 0 et delay
      jittered_delay = random.uniform(0, delay)

      print(
          f"  🔄 Tentative {attempt+1}/{max_retries} "
          f"échouée (status={status}). "
          f"Retry dans {jittered_delay:.1f}s…"
      )
      time.sleep(jittered_delay)

  return status, body


# ── Exemple d'utilisation ──
if __name__ == "__main__":

  # On crée une closure pour passer les paramètres
  def call_health():
      return api_request("GET",
                         "http://127.0.0.1:8080/health",
                         timeout=5)

  status, body = request_with_retry(call_health)
  print(f"Résultat final : {status} → {body}")
⚠️ Piège classique — Retry sur opérations non idempotentes

Le code ci-dessus est sûr pour des GET (lecture, idempotent). Mais ne retentez JAMAIS aveuglément un POST /payment ou POST /orders sans mécanisme d'idempotence.

Avant de retry un POST, demandez-vous :

  • Le serveur supporte-t-il un Idempotency-Key ?
  • La requête a-t-elle réellement échoué, ou est-ce seulement la réponse qui s'est perdue ?
  • Puis-je d'abord vérifier l'état (GET) avant de retenter ?

En cas de doute : ne pas retenter, remonter l'erreur et laisser l'humain décider.

Checklist « API Security Baseline »

Cette checklist minimale doit être vérifiée pour chaque API mise en production :

  • TLS/HTTPS activé sur tous les endpoints (y compris internes)
  • Authentification requise sur tous les endpoints sauf /health

  • Rotation des secrets planifiée (au moins tous les 90 jours)
  • Audit logs sur : login, échec login, création/suppression, accès admin
  • Pas de données sensibles dans les logs (tokens, mots de passe masqués)
  • Headers de sécurité : X-Content-Type-Options: nosniff, X-Frame-Options
  • UUIDs pour les identifiants de ressources (pas d'IDs séquentiels)
  • CORS configuré de manière restrictive (pas de * en production)

Étude d'incident : « Token leaked »

🔒 Scénario d'incident

Un développeur a accidentellement commité un fichier .env contenant le token de service du Document Service dans le dépôt Git public. Le token donne un accès complet en lecture/écriture à tous les documents. Le commit a été poussé il y a 3 heures et un collègue vient de s'en apercevoir.

Étapes de réponse à l'incident (dans l'ordre) :

Étape Action Pourquoi
Containment immédiat Révoquer le token compromis immédiatement (le rendre invalide côté Auth Service) Bloquer tout accès malveillant en cours. Chaque minute compte.
Rotation Générer un nouveau token de service et le déployer sur le Document Service Rétablir le fonctionnement normal avec des credentials sains
Audit Analyser les logs d'accès des 3 dernières heures : quelles requêtes ont utilisé ce token ? Depuis quelles IPs ? Quelles actions ? Déterminer si le token a été exploité par un tiers
Évaluation d'impact Si des accès suspects sont détectés : identifier les données touchées, notifier les utilisateurs concernés Obligation légale (RGPD) et éthique en cas de fuite de données
Nettoyage Supprimer le fichier .env du dépôt. Attention : un simple git rm ne suffit pas, le secret reste dans l'historique. Utiliser git filter-branch ou BFG Repo-Cleaner Le secret ne doit plus être accessible, même dans l'historique
Prévention Ajouter .env au .gitignore. Mettre en place un outil de détection de secrets dans le CI (pre-commit hooks, scanning) Empêcher que l'incident se reproduise
Post-mortem Rédiger un rapport d'incident : timeline, cause racine, impact, actions correctives, leçons apprises Culture d'amélioration continue (blameless post-mortem)
⚠️ Piège classique
« J'ai supprimé le commit, c'est bon. » — Non. Si le dépôt est public, des bots scannent GitHub en continu et peuvent avoir copié le secret en quelques secondes. La révocation immédiate du token est toujours la première action, avant même de nettoyer le dépôt.

Synthèse — À retenir en 10 points

# Point clé Catégorie
1 La communication synchrone (HTTP) crée un couplage temporel fort : les deux parties doivent être disponibles simultanément. Le messaging réduit ce couplage mais ajoute de la complexité. API
2 Une API est un contrat. Chaque endpoint doit avoir des méthodes, payloads, codes d'erreur et règles de validation documentés explicitement. API
3 L'idempotence est cruciale pour la fiabilité : si une opération peut être appelée N fois sans changer le résultat, les retries sont sûrs. Sinon, utilisez des Idempotency-Key. Fiabilité
4 Tout appel réseau doit avoir un timeout. Sans timeout, un service lent peut bloquer et contaminer l'ensemble du système (effet cascade). Fiabilité
5 Les retries doivent utiliser un backoff exponentiel avec jitter pour éviter les retry storms. Ne jamais retenter aveuglément des opérations non idempotentes. Fiabilité
6 Le circuit breaker protège un service malade : après N échecs, les requêtes sont rejetées immédiatement (fail fast), puis un test périodique vérifie la reprise. Fiabilité
7 AuthN ≠ AuthZ. L'authentification vérifie l'identité (qui es-tu ?), l'autorisation vérifie les droits (as-tu le droit ?). Les deux sont nécessaires et dans cet ordre. Sécurité
8 La validation d'entrée est la première ligne de défense. Toute donnée reçue de l'extérieur est potentiellement malveillante. Valider type, format, longueur, valeurs autorisées. Sécurité
9 « Internal network » ≠ confiance. Le Zero Trust impose de vérifier chaque requête, même entre services internes (mouvement latéral, insider threat). Sécurité
10 L'observabilité (logs structurés, traces, métriques, correlation IDs) est indispensable pour diagnostiquer les problèmes dans un système distribué. On ne peut pas debugger ce qu'on ne voit pas. Concept

Mini-Quiz — 10 questions + Corrigé

Q1. Quelle est la principale différence entre le couplage temporel d'un appel HTTP synchrone et d'un système de messaging ?
  • L'appel HTTP est plus rapide que le messaging
  • En HTTP synchrone, client et serveur doivent être disponibles simultanément ; en messaging, le producteur peut envoyer même si le consommateur est absent
  • Le messaging ne supporte pas le JSON
  • L'appel HTTP ne nécessite pas de réseau
✅ Réponse : B. En communication synchrone (HTTP), le client bloque en attendant la réponse du serveur — les deux doivent être actifs au même moment (couplage temporel fort). En messaging, le producteur dépose un message dans un broker et continue ; le consommateur le traite quand il est disponible (couplage temporel faible).
Q2. Un client envoie POST /api/v1/orders pour créer une commande. Le réseau est lent et le client ne reçoit pas de réponse (timeout). Le client retente automatiquement la même requête. Que risque-t-il de se passer ?
  • Rien, POST est idempotent
  • La commande est créée deux fois (duplication)
  • Le serveur rejette automatiquement la duplication
  • Le timeout est remis à zéro
✅ Réponse : B. POST n'est pas idempotent par défaut. Le serveur a potentiellement traité la première requête avec succès (mais la réponse s'est perdue). Le retry crée une deuxième commande. Solution : utiliser un Idempotency-Key pour que le serveur détecte la duplication.
Q3. Pourquoi le backoff exponentiel inclut-il un composant « jitter » (aléatoire) ?
  • Pour accélérer les retries
  • Pour empêcher que tous les clients retentent exactement au même instant (retry storm)
  • Pour rendre le code plus complexe et sécurisé
  • Pour respecter les normes HTTP
✅ Réponse : B. Sans jitter, si 1000 clients subissent un timeout en même temps et utilisent le même backoff (1s, 2s, 4s…), ils retenteront tous au même instant, créant un pic de charge. Le jitter étale les retries dans le temps, réduisant la pression sur le serveur.
Q4. Quelle est la différence entre une erreur 401 Unauthorized et 403 Forbidden ?
  • 401 = authentifié mais pas autorisé ; 403 = pas authentifié
  • 401 = pas authentifié (ou token invalide) ; 403 = authentifié mais pas les droits
  • 401 = erreur serveur ; 403 = erreur client
  • Aucune différence, les deux sont interchangeables
✅ Réponse : B. 401 signifie que l'authentification a échoué ou est manquante (AuthN). 403 signifie que l'utilisateur est bien identifié mais n'a pas la permission d'effectuer l'action demandée (AuthZ).
Q5. Un circuit breaker est dans l'état « OUVERT ». Que se passe-t-il quand le client tente un appel ?
  • La requête est envoyée normalement au serveur
  • La requête est mise en file d'attente
  • La requête est immédiatement rejetée sans être envoyée (fail fast)
  • La requête est envoyée en double pour vérifier
✅ Réponse : C. Quand le circuit breaker est ouvert, il court-circuite l'appel : la requête est rejetée immédiatement, sans consommer de ressources réseau. Après un délai défini, il passe en « semi-ouvert » et laisse passer une requête de test.
Q6. Pourquoi ne faut-il jamais retenter une erreur 400 Bad Request ?
  • Parce que 400 signifie que le serveur est en panne
  • Parce que la requête est invalide : la retenter identiquement donnera toujours le même résultat
  • Parce que le rate limiting l'interdit
  • Parce que 400 est un code de succès partiel
✅ Réponse : B. Un code 400 indique que la requête elle-même est malformée (champs manquants, format invalide). Envoyer la même requête N fois ne changera rien. Les retries ne sont utiles que pour les erreurs transitoires (5xx, problèmes réseau).
Q7. Qu'est-ce que le SSRF (Server-Side Request Forgery) dans le contexte d'une API ?
  • Un type de chiffrement pour les communications internes
  • Une attaque où l'attaquant fait exécuter des requêtes par le serveur vers des ressources internes auxquelles il n'a pas accès directement
  • Un protocole de transfert de fichiers sécurisé
  • Un mécanisme de rate limiting
✅ Réponse : B. Le SSRF exploite un serveur qui accepte des URLs en entrée (ex : « télécharger un fichier depuis cette URL »). L'attaquant fournit une URL interne (http://169.254.169.254/metadata sur le cloud, ou http://localhost:6379 pour Redis) et le serveur exécute la requête avec ses propres privilèges réseau. Contre-mesure : valider et filtrer les URLs côté serveur, bloquer les IPs privées.
Q8. Quel est le rôle d'un X-Request-Id (ou correlation ID) dans un système distribué ?
  • Authentifier l'utilisateur
  • Chiffrer les données en transit
  • Permettre de suivre une requête à travers tous les services qu'elle traverse pour le debugging
  • Limiter le nombre de requêtes par seconde
✅ Réponse : C. Le X-Request-Id est un identifiant unique généré à l'entrée du système et propagé à chaque service. Il permet de retrouver dans les logs tous les traitements liés à une même requête utilisateur, ce qui est essentiel pour le debugging distribué.
Q9. Un développeur a commité un token d'API dans un dépôt Git public et s'en aperçoit 2 heures plus tard. Quelle est la première action à effectuer ?
  • Supprimer le fichier du dépôt avec git rm
  • Réécrire l'historique Git avec git filter-branch
  • Révoquer (invalider) immédiatement le token compromis
  • Changer le mot de passe de son compte GitHub
✅ Réponse : C. La priorité absolue est de révoquer le token pour bloquer tout accès malveillant immédiat. Le nettoyage du dépôt est important mais secondaire : des bots scannent les dépôts publics en continu et peuvent avoir copié le secret en quelques secondes.
Q10. Pourquoi le principe Zero Trust considère-t-il que le « réseau interne » n'est pas suffisant comme périmètre de sécurité ?
  • Parce que le réseau interne est toujours plus lent que l'externe
  • Parce qu'un attaquant ayant compromis un seul service interne peut pivoter librement vers tous les autres si aucune authentification inter-services n'est en place
  • Parce que le Zero Trust interdit l'utilisation de réseaux privés
  • Parce que les firewalls sont obsolètes
✅ Réponse : B. Le modèle de sécurité périmétrique (« château fort ») suppose que tout ce qui est à l'intérieur du réseau est sûr. En réalité, un attaquant qui pénètre le périmètre (via phishing, supply chain, vulnérabilité) peut se déplacer latéralement sans obstacle. Le Zero Trust impose de vérifier chaque requête, même interne, avec authentification mutuelle (mTLS), autorisation et logs.

Barème d'évaluation

Critère Poids Indicateurs de qualité
Qualité du contrat d'API 35 %
  • Endpoints cohérents et complets (tous les cas d'usage couverts)
  • Méthodes HTTP correctes (sémantique respectée)
  • Codes d'erreur pertinents et différenciés
  • Payloads bien définis (types, contraintes, exemples)
  • Idempotence identifiée et gérée
  • Pagination et versioning documentés
Cohérence des politiques de fiabilité 35 %
  • Timeouts réalistes et justifiés par service
  • Politique de retry adaptée au type d'opération (lecture vs écriture)
  • Backoff + jitter correctement dimensionnés
  • Scénarios analysés avec rigueur (risque + politique + justification)
  • Situations de fallback identifiées
Analyse de sécurité 30 %
  • Trois surfaces d'exposition correctement cartographiées
  • Menaces pertinentes et réalistes identifiées
  • Contrôles proposés techniquement viables
  • Priorités cohérentes (critique vs modéré)
  • Mécanisme de token décrit (génération, transmission, vérification, invalidation)
📌 À retenir — Critères transversaux
  • Cohérence : les trois livrables doivent être cohérents entre eux (les mesures de sécurité du L3 doivent correspondre aux annotations sécurité du L1)
  • Justification : chaque choix doit être argumenté (pas de réponse « parce que c'est une bonne pratique »)
  • Réalisme : les valeurs (timeouts, seuils de rate limiting, nombre de retries) doivent être réalistes pour un système de gestion documentaire
