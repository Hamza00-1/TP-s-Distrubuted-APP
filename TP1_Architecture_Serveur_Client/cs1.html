Analyse de l'Architecture Serveur-Client HTTP (REST/JSON)

Réalisé par : Hamza khchichine (Groupe 1)

Introduction

Ce rapport présente l'analyse des travaux pratiques portant sur la mise en place d'une communication client-serveur classique sous le protocole HTTP. Nous avons étudié et corrigé deux scripts en Python : un serveur HTTP minimaliste fournissant une API REST (Server.py), et un client HTTP requêtant cette API avec une gestion des temps d'attente réseau (client.py).

1. Analyse du Serveur (Server.py)

Le script Server.py utilise la bibliothèque standard http.server de Python pour implémenter un serveur web sur l'adresse 127.0.0.1:8000.

Fonctionnalités Clés :

  • Simulation de Base de Données : Une structure de données en mémoire (dictionnaire) est utilisée pour simuler une collection de documents.
  • API REST JSON : Le serveur répond aux requêtes GET sur les routes /documents (pour lister tous les documents) et /documents/{id} (pour récupérer un document spécifique). Il renvoie les données au format application/json.
  • Simulation de Latence Réseau : Pour reproduire les conditions du monde réel (comme la charge réseau ou serveur), le serveur introduit un délai artificiel aléatoire (via time.sleep()) pouvant aller de 0 à 2 secondes avant de répondre.

Observations issues de la navigation Web (Capture 1) :

La première capture d'écran montre le résultat de la requête GET sur http://127.0.0.1:8000/documents depuis Mozilla Firefox (ou interface web de type visualiseur JSON). Le serveur a bien répondu avec un flux JSON correctement formaté représentant l'intégralité des documents, avec une coloration syntaxique des structures : les clés, les valeurs numériques (id) et les chaînes de caractères (titre et auteur des documents comme "Rapport Q3" par "Alice").

2. Analyse du Client (client.py)

Le script client.py exploite le module natif urllib.request pour interagir avec le serveur. L'objectif était d'appeler plusieurs fois l'API afin de visualiser la gestion de la latence de manière client.

Fonctionnalités Clés :

  • Requêtes répétées : Le client demande le document d'ID 1 à 5 reprises au cours d'une boucle.
  • Fixation d'un délai d'attente (Timeout) : Les requêtes réseau peuvent bloquer l'exécution indéfiniment si le serveur ne répond jamais. Le client intègre un mécanisme de sécurité (TIMEOUT_SECONDS = 1.5) déclenchant une exception réseau si le serveur tarde.
  • Gestion d'Exceptions Renforcée : Afin d'éviter un plantage, le client a été corrigé pour intercepter correctement le TimeoutError (qui remonte depuis le socket Python bas-niveau) ainsi que l'urllib.error.URLError.

3. Analyse des Résultats d'Exécution Terminal (Capture 2 & Console)

La deuxième capture et nos exécutions montrent les logs générés par le client dans un terminal intégré de type VS Code. L'encodage a par ailleurs été corrigé pour empêcher la console de générer des conflits d'unicode sous Windows (les emojis et accents ont été retirés).

Plusieurs scénarios se présentent aléatoirement pour chaque requête à cause de la latence simulée côté serveur :

Reponse recue en 1.10s
Donnees : {'id': 1, 'titre': 'Rapport Q3', ...}

TIMEOUT apres 1.60s - le serveur est trop lent !
  • Requêtes rapides (Latence < 1.5s) : Le client reçoit la réponse à temps, décode correctement le JSON en dictionnaire Python, et affiche avec succès le document reçu en mesurant le temps exact.
  • Requêtes lentes (Latence > 1.5s) : Le serveur bloque au-delà du timeout accordé au client. Le mécanisme limitatif de timeout=1.5 coupe la connexion locale de façon fiable et affiche fièrement : TIMEOUT apres [X]s.
  • Remarque d'entrelacement : Sur la deuxième capture d'écran de l'affichage terminal, on observe que certaines informations se mélangent parfois sur la même ligne (ex: "...Rapport Q3'...TIMEOUT apres..."). Il s'agit d'un problème d'entrelacement (buffering) lié aux logs émis simultanément depuis les différents scripts (client/serveur) dans le même terminal virtuel.

4. Analyse de l'Exemple 2 : Retry avec Backoff Exponentiel

Le script client_retry.py implémente un client HTTP robuste capable de relancer une requête échouée de manière espacée.

Fonctionnalités Clés :

  • Idempotence et Retry : Lors d'une erreur (timeout ou réseau), le client effectue jusqu'à 4 tentatives.
  • Backoff Exponentiel avec Jitter : Le temps d'attente entre chaque échec est doublé (0.5s, 1s, etc.) et couplé à une valeur aléatoire (jitter) pour éviter de surcharger globalement le serveur.

Observations de l'exécution sur Terminal :

Appel avec retry + backoff exponentiel :
Tentative 1/4...
Echec : <urlopen error [WinError 10061]...>
Attente 0.62s avant retry...
Tentative 2/4...
Echec : TimeoutError
Attente 1.15s avant retry...
Tentative 3/4...
Succes a la tentative 3

Document recupere : {'id': 2, 'titre': 'Plan stratigique', 'auteur': 'Bob'}

Le terminal montre clairement l'escalade du temps d'attente généré aléatoirement avant un succès, stabilisant les échanges entre le client et un serveur défaillant.

5. Analyse de l'Exemple 3 : Appels Asynchrones

Le script client_async.py démontre l'intérêt de la parallélisation des requêtes réseau grâce au module asyncio.

Fonctionnalités Clés :

  • Programmation par Coroutines : Simulation de l'appel de multiples microservices en attente d'IO (Auth, Stockage, Recherche) via async/await.
  • Gain de latence réseau : Contraste frappant entre une boucle chronologique (séquentielle) et une exécution groupée via asyncio.gather().

Observations de l'exécution sur Terminal :

-- Appels SEQUENTIELS --
Appel a Auth (latence prevue : 0.5s)... Auth a repondu !
Appel a Stockage (latence prevue : 0.8s)... Stockage a repondu !
Appel a Recherche (latence prevue : 0.3s)... Recherche a repondu !
Temps total sequentiel : 1.61s

-- Appels PARALLELES (asyncio.gather) --
Appel a Auth (latence prevue : 0.5s)...
Appel a Stockage (latence prevue : 0.8s)...
Appel a Recherche (latence prevue : 0.3s)...
Recherche a repondu !
Auth a repondu !
Stockage a repondu !
Temps total parallele : 0.80s

Comme le montrent les logs, les appels parallèles abaissent considérablement le temps de réponse total puisqu'il correspond uniquement au maximum des latences individuelles (0.8s) et non plus à la somme de toutes les latences enchaînées (1.61s).

Conclusion Générale

Ce TP démontre efficacement les fondements des architectures Micro-services / API REST. La mise en œuvre illustre la découple entre un backend qui héberge la ressource et un backend/client qui la consomme.

Par-dessus tout, l'accent a été mis sur la gestion d'erreurs cruciales : la résolution de problèmes d'échappement (Unicode et indentation Python) et, surtout, la gestion proactive des timeouts. Fixer des limites de temps aux requêtes est indispensable pour concevoir des applications résilientes, permettant à un client de ne pas bloquer complètement si l'un de ses services externes vient à ralentir dramatiquement.
